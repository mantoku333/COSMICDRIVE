# chumini 技術資料（読みやすさ優先版）

この資料は「30ページ固定」をやめて、要点ごとに理解できる章構成に再編した版です。  
各コード抜粋は、**日本語コメント付き**で意図が分かる形にしています。

## 1. この作品で重視した設計思想

### 何を最重要にしたか
運用・販売されているリズムゲームをプログラマー目線で分析すると、最も重要なのは「1曲追加するための実装コスト」です。  
追加のたびにコード変更が必要な設計は、楽曲数が増えるほど運用負債になります。

本作では、以下を軸に設計しています。
- 楽曲追加時にコードを書き換えない
- 譜面側でイベント（スキル、終了通知など）を配置できる
- 高密度譜面でも判定・描画性能を落とさない
- 運用に必要な周辺機能（設定、保存、文字描画）を内製する

## 譜面読み込み技術資料（1から）

### この章の目的
この章は、譜面データがどのようにゲーム内部へ取り込まれ、最終的に判定可能な `NoteData` 列へ変換されるかを、コードを追わなくても理解できる形で説明するためのものです。  
対象範囲は、選曲画面での曲検出、ヘッダ先読み、プレイ開始時のフルパース、テンポマップ生成、ノーツ時刻計算、実行系への受け渡しまでです。  
要点は「読み込みを2段階に分離して、表示速度とプレイ精度を両立する」設計にあります。

### SUSファイルとは（本プロジェクトでの定義）
SUSは、**テキストベースで譜面・テンポ・メタ情報を同居させる譜面フォーマット**として扱っています。  
このプロジェクトでは、拡張子 `.sus` と `.ched` を同じパーサ（`ChedParser`）で読み込んでおり、実際の解釈は「ChedParserが対応している構文集合」に準拠します。  
つまり、SUS全仕様を網羅するというより、**運用に必要なサブセットを安定して処理する**方針です。

```txt
This file was generated by Ched 3.2.0.0.
#TITLE "BPMTEST"
#ARTIST "mantoku"
#WAVE "BPMtest.wav"
#JACKET "BPMTest.png"
#REQUEST "ticks_per_beat 480"
#00002: 4
#BPM01: 150
#00008: 01
#00010: 14141414
#0161c: 00440000
```
出典: `Songs/TEST/BPMTest/BPMTest.sus:1-47`

### SUSの構造（読み取り順）
1. ヘッダ行（`#TITLE`, `#ARTIST` など）  
曲情報、難易度、音源ファイル、ticks_per_beat などのメタ情報を保持します。
2. BPM定義行（`#BPMxx: value`）  
テンポID（`xx`）と実BPM値を対応付けます。後段の `#mmm08` から参照されます。
3. スコア本文行（`#mmmcc:data` / `#mmmccc:data`）  
小節番号 + チャンネル + データ列で、ノーツ配置やイベント配置を表現します。

### 1行フォーマットの意味
- `mmm`: 3桁小節番号（例: `016`）  
- `cc` または `ccc`: チャンネル識別子（例: `08`, `10`, `20A`）  
- `data`: 2文字単位トークン列（例: `14141414`, `00440000`）  
この実装では `data` を2文字ずつ切って時系列へ展開し、`00` は「イベントなし」としてスキップします。

### 図: dataトークンの時間分割（ASCII）
```txt
data = 14141414   (長さ8文字 = 2文字トークン x 4)

 index:      0      1      2      3
 token:    [14]   [14]   [14]   [14]
 time :     |------|------|------|------|  (1小節内を4分割)

data = 00440000

 index:      0      1      2      3
 token:    [00]   [44]   [00]   [00]
 meaning:   空    SongEnd  空     空
```

### トークンの基本解釈（本実装）
- 2文字トークンは base36 前提（`0-9A-Za-z`）で数値化可能。  
- `00`: 空イベント（無視）  
- `44`: SongEnd（楽曲終了通知）  
- `24` / `2g`: Skill ノーツ  
- その他（1n系）: Tap系として処理（`token[1] <= 0` は除外）

### チャンネル解釈（本実装で使用）
- `#mmm02`: 小節長（拍数）  
- `#BPMxx`: BPMテーブル定義  
- `#mmm08`: BPM変更イベント（`xx` 参照）  
- `#mmm1n`: 通常ノーツ系（Tap/Skill/SongEnd）  
- `#mmm2xy`, `#mmm3xy`, `#mmm4xy`: ロング/スライド系（`x`=レーン, `y`=holdId）

### 注意点（資料として明示すべき仕様境界）
- `ChedParser` は未知のヘッダを基本的に無視し、必要項目だけ取得します。  
- `.sus` と `.ched` は同一ロジックで処理されるため、差異は拡張子ではなく記述内容に依存します。  
- ロング系は生パース段階で `isHold=true` として収集し、最終的な開始/終了ペアは `ChartLoader` で再構築します。  
- 一般的なSUS仕様にある全チャンネル/全機能を網羅しているわけではなく、**このゲームで使う機能に絞って実装**しています。

### 実ファイルにあるが本実装で未使用の代表例
- `#DESIGNER`, `#SONGID`, `#WAVEOFFSET` は読み飛ばし（保持しない）。  
- `#TILxx`, `#HISPEED`, `#MEASUREHS` などの拡張表現は、現在の判定・描画パイプラインでは未採用。  
- そのため、エディタ上で設定されていても本ゲームで意味を持つのは、実装済みヘッダ/チャンネルのみです。

### レーン正規化の補足（4K前提）
SUS側のレーン文字は base36 値として解釈し、`rawLane / 4` を `0..3` にクランプして4レーンへ正規化しています。  
このため、入力側レーン定義が細かく分かれていても、ゲーム内では4K用の共通レーン座標へ集約されます。

### 入力仕様（この章で前提とするデータ）
- 対象拡張子: `.ched` / `.sus`  
- ヘッダ系: `#TITLE`, `#ARTIST`, `#WAVE`, `#JACKET`, `#PLAYLEVEL`, `#DIFFICULTY`, `#REQUEST ticks_per_beat`, `#BPMxx`  
- スコア本文: `#mmmcc:data` 形式（`mmm`=小節, `cc`=チャンネル）  
- 主なチャンネル解釈:  
`08`=BPM変更, `1n`=通常系ノーツ, `2n/3n/4n`=ロング系ノーツ

### 全体フロー（先に結論）
1. `SongListService` が `Songs` 以下を走査し、譜面ファイルを発見する。  
2. `ChedParser::Load(..., true)` でヘッダのみ先読みし、曲一覧表示に必要な最小情報を作る。  
3. プレイ開始時に `NoteManager` が `ChartLoader::Load(...)` を呼び、フルロードを開始する。  
4. `ChartLoader` 内部で `ChedParser::Load(..., false)` を実行し、BPMイベントと生ノーツ列を取得する。  
5. `TempoService::BuildTempoMap` でテンポイベントを実時間計算可能な形へ前処理する。  
6. 生ノーツを `NoteData` へ正規化し、ホールドのペア再構築、`hittime` 計算、コンボ算出を行う。  
7. `ChartLoadResult` を `NoteManager` が受け取り、`noteSequence` / `tempoMap` として実行系へ反映する。

### フェーズ1: 選曲一覧のためのヘッダ先読み
選曲画面では、全譜面をフルパースせず、タイトル・アーティスト・BPMなどの表示情報だけを先に取得します。  
ここで重要なのは、プレイヤーが最初に触る画面の応答性を守ることです。  
もしこの段階でノーツ展開や時刻計算まで実行すると、楽曲数の増加に比例して起動直後の待ち時間が伸びます。  
本実装は `headerOnly=true` による軽量ロードを採用し、表示に不要な処理を後段へ明確に遅延させています。  
その結果、コンテンツ量が増えても選曲一覧の体感速度を維持しやすい構成になっています。

### フェーズ2: プレイ開始時のフルロード起点
実プレイに入るタイミングで、`NoteManager` は `ChartLoader` にフルロードを委譲します。  
この段階は「表示用データ」から「判定用データ」へ責務を切り替える境界です。  
同時に、譜面側オフセットとユーザー設定オフセットを引数として受け取り、時間軸の補正を一か所で確定します。  
これにより、譜面仕様由来の補正と個人設定由来の補正が分散せず、後続処理の一貫性が保たれます。

### フェーズ3: ChedParser 一次パス（行正規化とメタ抽出）
`ChedParser` は最初にファイルを1行ずつ読み、改行差分や空白、コメントを正規化します。  
その上で `#TITLE` / `#ARTIST` / `#REQUEST` などのメタ情報を取り込み、同時にスコア本文を `scoreLines` へ蓄積します。  
この時点ではノーツ確定処理をせず、「後段で解析しやすい中間形」に整理するのが目的です。  
先に入力テキストを正規化しておくことで、文字コード差や空白揺れの影響を後続ロジックへ持ち込まない設計になります。  
また、ヘッダ処理と本文蓄積を同じ走査で完了させるため、I/O回数を増やさずに必要情報を収集できます。  
この一次パスは、パース精度を上げるための前処理層として機能します。

```cpp
// ChedParser.cpp

while (std::getline(file, rawLine)) {

        if (cmd == L"#ARTIST") { artist = RemoveQuotesAndToUtf8(val); continue; }
        if (cmd == L"#JACKET") { jacketFile = RemoveQuotesAndToUtf8(val); continue; }
        //~~中略~~//
        if (cmd.find(L"#BPM") == 0) {
            try {
                double d = std::stod(val);
                // 初期BPMをInfoに投げる
                if (this->bpm == 0) {
                    this->bpm = static_cast<int>(d);
                }
            }
            catch (...) {}

            // フルロードする必要がない場合、BPMが取れた時点で満足して帰る
            if (headerOnly && this->bpm > 0) {
                return true;
        }
    }

    // 譜面データをバッファリング
    if (line[0] == L'#') {
        auto colonPos = line.find(L':');
        if (colonPos != std::wstring::npos) {
            std::wstring headW = line.substr(1, colonPos - 1);
            std::wstring dataW = line.substr(colonPos + 1);
            TrimW(dataW);
            dataW.erase(std::remove_if(dataW.begin(), dataW.end(), 
            [](wchar_t c) { return iswspace(c); }), dataW.end());

            if (!dataW.empty()) {
                std::string head(headW.begin(), headW.end());
                std::string data(dataW.begin(), dataW.end());
                scoreLines.emplace_back(std::move(head), std::move(data));
            }
        }
    }
}
```

### フェーズ4: 小節情報からグローバルtickへ変換
可変小節長（例: 3拍子区間）に対応するため、まず各小節の長さを収集し、累積tickを持つ `BarInfo` 配列を作ります。  
これにより、任意の譜面位置を「曲全体で単調増加するtick」に変換できます。  
以後のBPMイベント・ノーツ時刻計算はこのグローバルtick基準で行われます。  
この変換を先に確立することで、拍子変化を含む譜面でも時系列の整合を崩さず扱えます。  
特に、同一小節内の分割数が異なるケースでも、`ToGlobalTick` に統一することで比較・ソート・変換を共通化できます。  
結果として、後段は「何小節目か」ではなく「曲全体での位置」に集中でき、実装が単純になります。

```cpp
// ChedParser.cpp

// 小節長テーブル構築
std::vector<std::pair<int, double>> barLengths;
for (auto& h : scoreLines) {
    if (h.first.size() == 5 && h.first.substr(3, 2) == "02") {
        try {
            int measure = std::stoi(h.first.substr(0, 3));
            double beats = std::stod(h.second);
            barLengths.emplace_back(measure, beats);
        }
        catch (...) {}
    }
}

// 各小節の開始tickを累積計算
for (size_t i = 0; i < barLengths.size(); ++i) {
    int measure = barLengths[i].first;
    double beats = barLengths[i].second;
    if (i > 0) {
        // 前の小節区間の長さを足し合わせる
        int prevMeasure = barLengths[i - 1].first;
        double prevBeats = barLengths[i - 1].second;
        accumTicks += static_cast<int>((measure - prevMeasure) 
        * prevBeats * ticksPerBeat);
    }
    bars.push_back(BarInfo{ measure, static_cast<int>
    (beats * ticksPerBeat), accumTicks });
}

// 指定小節の情報を取得
auto findBar = [&](int measure) -> const BarInfo& {
    const BarInfo* current = &bars.front();
    for (auto& b : bars) {
        if (b.measureStart <= measure) current = &b; else break;
    }
    return *current;
};

// 曲頭からの秒数へ変換
auto ToGlobalTick = [&](int measure, int index, int total) -> int {
    const BarInfo& bar = findBar(measure);
    int baseTick = bar.startTick + (measure - bar.measureStart) 
    * bar.ticksPerMeasure;
    return baseTick + (index * bar.ticksPerMeasure) / total;
};


```

### フェーズ5: BPMテーブル構築とテンポイベント抽出
譜面中のBPM定義を `bpmTable` に集約し、08チャンネルのイベント列をテンポ変更として解釈します。  
各イベントは `absBeat`（絶対拍）で保持し、後段のテンポマップ構築に渡します。  
SUS本文ではBPM値そのものではなくID参照で書かれるため、先にテーブル化してからイベント展開する二段処理が必要です。  
この分離により、同じBPM値を複数箇所で再利用しても解釈が一貫し、譜面記述の重複も減らせます。  
テンポ情報をノーツ情報と同じ時間基準（absBeat）へ落とし込むのが、このフェーズの核心です。

```cpp
// ChedParser.cpp

    // 右辺の文字列を実BPM値として保存する
    bpmTable[id] = std::stod(h.second);

// #mmm08 をテンポ変更イベントとして展開
```

### フェーズ6: ノーツイベント展開（Tap/Skill/SongEnd/Hold）
通常ノーツは 1n チャンネル、ロング系は 2n/3n/4n で解析します。  
この段階では「譜面由来の生ノーツ列」を作ることに集中し、最終的なペア確定や時間確定は `ChartLoader` 側で実施します。  
また、`44`（SongEnd）や `24/2g`（Skill）を通常ノーツチャンネル内のコマンドとして解釈し、運用上必要なイベントを譜面へ埋め込めるようにしています。  
ロング系は `holdId` を保持したまま収集し、ここでは過剰に意味付けしない方針です。  
この「生データ化」フェーズを分離することで、後段での仕様変更に対して柔軟に追従できます。

### フェーズ9: テンポマップの意味（判定系との接続）
`TempoService` は「テンポイベント列を前処理して、拍と秒の変換を安価にする」ための層です。  
譜面ロード時に `BuildTempoMap` を1回実行しておくことで、プレイ中の変換は区間探索 + 局所計算のみで済みます。  
この設計は、判定処理・ホールドtick計算・演出同期など、複数サブシステムが同じ時間変換基盤を共有できる点が重要です。  
変換ロジックが一か所に集約されるため、可変BPM曲でのズレを修正する際も調整点が明確になります。  
結果として、精度と保守性を両立した時間基盤として機能します。

```cpp
// TempoService.cpp
void TempoService::BuildTempoMap(std::vector<TempoEvent>& tm) const {
    if (tm.empty()) {
        tm.push_back({ 0.0, DEFAULT_BPM, 60.0 / DEFAULT_BPM, 0.0 });
        return;
    }
    std::sort(tm.begin(), tm.end(),
        [](const TempoEvent& a, const TempoEvent& b) { return a.atBeat < b.atBeat; });

    tm[0].spb = 60.0 / tm[0].bpm;
    tm[0].atSec = 0.0;
    for (size_t i = 1; i < tm.size(); ++i) {
        tm[i].spb = 60.0 / tm[i].bpm;
        double beatsSpan = tm[i].atBeat - tm[i - 1].atBeat;
        tm[i].atSec = tm[i - 1].atSec + beatsSpan * tm[i - 1].spb;
    }
}
```

### フェーズ別 技術メモ（深掘り・抜粋用）

#### フェーズ1 技術メモ（ヘッダ先読み）
入力: `Songs` 配下のファイルパス列、拡張子フィルタ（`.sus` / `.ched`）。  
出力: 選曲一覧に必要な `SongInfo` の最小集合（title/artist/chartPath など）。  
不変条件: 一覧構築段階では「ノーツ展開をしない」。`parser.Load(..., true)` を通すことで、表示に不要な処理を混入させない。  
失敗時挙動: 個別ファイルの読み込みに失敗しても、他ファイルの走査は継続する。曲が1件も見つからない場合はダミー曲を返してUIを壊さない。  
計算量の見積もり: フォルダ走査に対して線形。フルロードを遅延するため、楽曲数増加時の体感遅延を抑えやすい。

#### フェーズ2 技術メモ（フルロード開始点）
入力: `chartPath`, `K_OFFSET_SEC`, `gGameConfig.offsetSec`。  
出力: `ChartLoadResult` を `noteSequence`, `tempoMap`, `songEndIndex`, `maxTotalCombo` へ移送した実行準備済み状態。  
不変条件: 時間補正の責務をロード境界に集約し、判定ループ中に補正値を再解釈しない。  
失敗時挙動: チャート読み込み失敗時は空に近い `ChartLoadResult` が返る設計で、呼び出し側は異常系を明確に検出しやすい。  
運用上の利点: 「どこでゲーム用データが確定するか」が明確なので、再現調査ログの取得位置を固定できる。

#### フェーズ3 技術メモ（一次パス）
入力: 生テキスト行（改行差分、コメント、空白、文字コード揺れを含む）。  
出力: メタ情報（title/artist/ticksPerBeat 等）と、後段解析用 `scoreLines`（head/data）中間表現。  
不変条件: data部は空白除去済み、2文字トークン分割可能な連続文字列として保持する。  
失敗時挙動: `stoi/stod` 変換失敗は例外吸収し、該当項目のみ無視して処理継続する。全体を止めないことで不良行への耐性を持たせる。  
実装意図: 解析を1回のI/O走査で終わらせることで、速度と堅牢性のバランスを取っている。

#### フェーズ4 技術メモ（グローバルtick化）
入力: 小節長定義（`#mmm02`）と `ticks_per_beat`。

失敗時挙動: 小節長定義が存在しない譜面は `4/4` をデフォルト挿入して処理を継続する。  
計算量の見積もり: 小節情報の前処理は小節定義数に線形。後段の位置計算は軽量な算術演算のみ。

入力: `#BPMxx` 定義群 + `#mmm08` 参照イベント。

出力: `BeatToSeconds` / `SecondsToBeat` を高速に呼べる前処理済みテンポマップ。  
不変条件: 変換基盤を1か所へ集約し、判定・演出・ホールド計算で同一ルールを共有する。  
失敗時挙動: 空テンポ時はデフォルト120 BPMを適用し、ゼロ除算や未定義変換を防止する。  
計算量の見積もり: 前処理はソートを含むため `O(N log N)`、変換呼び出しは二分探索で `O(log N)`。

### 譜面読み込みの設計上の利点
- 一覧表示時はヘッダ先読みのみなので、曲数増加に対して初期表示が重くなりにくい。  
- フルロードは `ChartLoader` に集約され、ノーツ時刻計算・ペア再構築・コンボ算出の責務が明確。  
- 可変BPM対応が `absBeat + tempoMap` で統一され、判定や演出側が同じ時間基準を参照できる。  
- 譜面仕様の拡張（新しいコマンドノーツ追加）を、パーサ層の変更中心で吸収しやすい。

### 拡張時の実装ガイド（新しいノーツ種別を追加する場合）
1. `NoteType` に新種別を追加する。  
2. `ChedParser` のトークン判定へ新種別の割り当てを追加する。  
3. `ChartLoader` のコンボ算出対象に含めるかを決める。  
4. `NoteManager` の描画分岐・判定分岐・SE再生分岐へ反映する。  
5. 資料側では「譜面記法」「判定挙動」「コンボ加算条件」をセットで更新する。

## 4. 可変BPMを実時間へ落とし込むテンポマップ

### 設計意図
この章の目的は、可変BPM曲で頻発する「拍 <-> 秒」変換を実行時に重くしないことです。  
ノーツ出現、判定、演出同期の各タイミングで変換が何度も走るため、毎回先頭から積分するとCPUコストが増えます。  
そこでロード時にテンポイベントを時系列整列し、各区間の開始秒を前計算したテンポマップを構築します。  
実行中は現在拍に対応する区間を二分探索で引き、局所計算だけで秒へ変換できるようにしています。  
前計算型に寄せることで、譜面密度が上がっても時間変換の負荷が増えにくい構造にしています。

### コード抜粋
```cpp
// TempoService.cpp
void TempoService::BuildTempoMap(std::vector<TempoEvent>& tm) const {
    if (tm.empty()) {
        // BPM情報がない譜面はデフォルト120 BPMを適用
        tm.push_back({ 0.0, DEFAULT_BPM, 60.0 / DEFAULT_BPM, 0.0 });
        return;
    }

    // 絶対拍でソートして時間順を保証
    std::sort(tm.begin(), tm.end(), [](const TempoEvent& a, const TempoEvent& b) {
        return a.atBeat < b.atBeat;
    });

    // 各イベントの開始秒を累積計算
    tm[0].spb = 60.0 / tm[0].bpm;
    tm[0].atSec = 0.0;
    for (size_t i = 1; i < tm.size(); ++i) {
        tm[i].spb = 60.0 / tm[i].bpm;
        const double beatsSpan = tm[i].atBeat - tm[i - 1].atBeat;
        tm[i].atSec = tm[i - 1].atSec + beatsSpan * tm[i - 1].spb;
    }
}
```

## 6. Raw Inputによる低遅延入力

### 設計意図
この章の目的は、入力経路を短くして判定遅延を最小化し、改善効果を数値で検証できるようにすることです。  
標準のメッセージ処理だけに頼ると、OS側の処理順序やアプリ側キュー状況の影響を受けやすくなります。  
そこでRaw Input APIを利用し、HIDから届く入力情報を直接受け取り、内部キー状態へ即時反映します。  
さらにQPC（高精度タイマ）で入力受信時刻との差分をマイクロ秒単位でログ化し、設定変更の効果を比較可能にしています。  
体感だけでなく計測値でも改善を追える点が、運用段階での品質維持に効く設計です。

### コード抜粋（Raw Input登録と処理）
```cpp
// SInput.cpp
void SInput::InitRawInput(HWND hwnd)
{
    RAWINPUTDEVICE Rid[1];
    Rid[0].usUsagePage = 0x01;
    Rid[0].usUsage = 0x06; // キーボード
    Rid[0].dwFlags = 0;
    Rid[0].hwndTarget = hwnd;

    // Raw Inputデバイスを登録
    RegisterRawInputDevices(Rid, 1, sizeof(Rid[0]));
}

void SInput::ProcessRawInput(LPARAM lParam)
{
    // 1) まず必要バイト数だけ問い合わせる（実データはまだ受け取らない）
    UINT dwSize = 0;
    GetRawInputData((HRAWINPUT)lParam, RID_INPUT, NULL, &dwSize, sizeof(RAWINPUTHEADER));

    // 2) 取得サイズ分の一時バッファを確保する
    std::vector<BYTE> lpb(dwSize);

    // 3) 実データをバッファへ読み込む
    //    戻り値が要求サイズと一致しなければ不正扱いで終了
    if (GetRawInputData((HRAWINPUT)lParam, RID_INPUT, lpb.data(), &dwSize, sizeof(RAWINPUTHEADER)) != dwSize)
        return;

    // 4) 汎用RAWINPUT構造体として解釈する
    RAWINPUT* raw = (RAWINPUT*)lpb.data();

    // 5) 今回はキーボード入力のみ処理対象
    if (raw->header.dwType == RIM_TYPEKEYBOARD)
    {
        // 仮想キーコード（VK_*）を取得
        USHORT vKey = raw->data.keyboard.VKey;

        // RI_KEY_BREAK が立っていない = 押下、立っている = 離上
        // （Raw Inputのフラグ定義に基づく判定）
        bool isDown = !(raw->data.keyboard.Flags & RI_KEY_BREAK);

        // 6) 内部キー状態テーブルを直接更新する
        //    Windows標準のキーリピート処理を挟まず、受け取った生入力をそのまま反映
        if (isDown) SetKeyDown(vKey);
        else SetKeyUp(vKey);
    }
}
```

## 9. 判定探索の局所化（レーン head + n件先比較）

### 設計意図
この章の狙いは、入力判定処理の探索範囲を常に局所化して、譜面総ノーツ数の影響を受けにくくすることです。  
毎入力で全ノーツを走査する方式は、曲が長くなるほど無駄比較が増え、入力密度の高い場面で重くなります。  
そこでレーンごとに先頭位置（head）を保持し、判定済みを順次捨てながら、未判定先頭からn件だけを候補にします。  
さらに未スポーン領域に到達した時点で探索を打ち切り、未来ノーツへの不要アクセスを防ぎます。  
この設計により、判定計算量を実プレイに必要な最小範囲へ閉じ込め、スケーラビリティを確保しています。

局所化のために使っている主な状態は4つです。  
1. `laneOrder[lane]`: そのレーンに属するノーツindex列（時系列順）。  
2. `laneHeads[lane]`: 次に判定候補として見る先頭位置。  
3. `nextIndex`: 既にスポーン済みの末尾境界（未来ノーツを判定対象外にする）。  
4. `K_LOOKAHEAD_NOTES`: 1入力あたりに先読みする最大件数。  

この4つを固定すると、判定対象は「対象レーンの先頭近傍 + スポーン済み範囲」に限定されます。  
全体配列を毎回なめる必要がなくなるため、譜面密度が上がっても入力1回あたりの処理時間を安定化できます。  
特に120FPS運用では、1フレーム内に複数入力が重なる場面があるため、この上限管理が効きます。

### フェーズ1: レーン索引を先に構築しておく
入力判定の高速化は、実はロード直後の前処理でほぼ決まります。  
`noteSequence` をそのまま検索対象にせず、レーン別の索引 `laneOrder` を作ることで、入力時の探索範囲を最初から分離します。

```cpp

// レーンごとの走査先頭（head）を0で初期化

    // ノーツのlane値を安全な範囲へ丸める
    int l = std::clamp(noteSequence[i].lane, 0, lanes - 1);
    // 「レーンlに属するノーツindex i」を時系列順で登録
    laneOrder[l].push_back(i);
}

```

### フェーズ2: 入力時は head から最大 n 件だけを候補収集
実入力時の `JudgeLane` では、`laneHeads[lane]` から先頭を開始し、最大 `K_LOOKAHEAD_NOTES` 件だけを見る実装です。  
このとき、判定済み・曲終端・非アクティブはその場で捨て、先頭であれば `head` を前進させて再訪を防ぎます。  
また `idx >= nextIndex` に到達したら「まだ画面に出していない未来ノーツ」とみなし、その時点で探索を打ち切ります。

### コード抜粋（候補収集ループ）
```cpp
// JudgeConstants.h
inline constexpr int K_LOOKAHEAD_NOTES = 5; // 先読み件数の上限

// NoteManager.cpp (JudgeLane)
// このレーンに属するノーツindex列
auto& order = laneOrder[lane];
// このレーンの探索開始位置（判定が進むと前進していく）
size_t& head = laneHeads[lane];

std::vector<JudgeCandidate> candidates;
// 先読み上限が決まっているので、再確保を避けるために予約
candidates.reserve(K_LOOKAHEAD_NOTES);

// 先頭位置から探索を開始
size_t currentIdx = head;
// 収集した候補数
int count = 0;
// 1入力で走査する最大件数（上限を固定）
const int MAX_SCAN = K_LOOKAHEAD_NOTES;

// 末尾到達 or 先読み上限到達まで探索
while (currentIdx < order.size() && count < MAX_SCAN) {
    // 実際のノーツ配列上のindex
    int idx = order[currentIdx];

    // 判定済み/終端は候補に入れない。先頭ならheadを進めて再探索を削減
    if (noteSequence[idx].judged || noteSequence[idx].type == NoteType::SongEnd) {
        // 先頭が無効だった場合はhead自体を前進（次回から見ない）
        if (currentIdx == head) ++head;
        ++currentIdx;
        continue;
    }

    // 未来ノーツ（未スポーン）に達したら、以降も未来なので探索終了
    if (idx >= (int)nextIndex) break;

    // 対応アクターが既に無効なら候補から除外
    auto* act = noteActors[idx].Target();
    if (!act) {
        if (currentIdx == head) ++head;
        ++currentIdx;
        continue;
    }

    // タップ判定対象外の種別はここで除外
    if (noteSequence[idx].type == NoteType::HoldEnd ||
        noteSequence[idx].type == NoteType::Skill) {
        if (currentIdx == head) ++head;
        ++currentIdx;
        continue;
    }

    // 判定に使う見た目上のZ位置
    float noteZ = act->transform.GetPosition().z;
    // HoldStartは長方体の先端位置を判定に使いたいので前方へ補正
    if (noteSequence[idx].type == NoteType::HoldStart) {
        float slopeRad = rotX * 3.14159265f / 180.0f;
        float halfLenZ = act->transform.GetScale().z * std::cos(slopeRad) * 0.5f;
        noteZ -= halfLenZ;
    }

    // JudgeServiceへ渡すため、必要最小の情報だけを候補へ詰める
    JudgeCandidate c;
    c.noteIndex = idx;
    c.type = noteSequence[idx].type;
    c.hittime = noteSequence[idx].hittime;
    c.zPosition = noteZ;
    candidates.push_back(c);

    // 候補を1件積んだので進行
    ++count;
    ++currentIdx;
}
```

### フェーズ3: 候補から1件を確定し、判定を返す
候補収集後は `JudgeService::JudgeCandidates` に委譲し、時間差判定と位置許容（Z方向）を組み合わせて1件を選びます。  
ここでのポイントは、早すぎる入力に対しては `Skip` を返して次の入力機会へ回す設計です。  
これにより、未来ノーツを誤って巻き取るリスクを下げています。

### コード抜粋（候補確定ロジック）

```cpp
    // 入力時刻 - ノーツ基準時刻（+なら遅い、-なら早い）

    // 早すぎる入力は確定せず終了（次フレーム/次入力で再判定）
    if (diff < -K_JUDGE_GOOD) return action; // Skip

        else continue;               // 遅い側はこの候補を捨てて次候補を試す
    }
```

### 計算量と運用上の意味
1. 1入力あたりの候補探索は概ね `O(K_LOOKAHEAD_NOTES)` に制限される。

## 11. バイナリセーブ（設定 + スコア）

### 設計意図
この章の狙いは、起動時や画面遷移時に頻繁に発生する保存/読込を軽量化し、運用で壊れにくい永続化基盤を持つことです。  
本実装は保存対象を2系統に分離しています。  
1. `Config` 系: キー設定やオプションを `Save/Save.bin` へ保存。  
2. `Score` 系: 譜面ごとのハイスコアを `Save/Scores.bin` へ保存。  

この分離により、設定変更とスコア更新が互いのフォーマットに影響しにくくなります。  
また、Configは固定長構造体、Scoreは可変長エントリという方針を使い分けています。  
Configは実装を短くでき、Scoreは拡張フィールド（difficulty/rating/title）を段階的に追加できます。  

### フェーズ0: コンフィグ機能の紹介（バイナリセーブとの接続）
ここでは保存処理に入る前に、「何を保存対象にしているか」を先に定義します。

1. プレイ調整: `HiSpeed`, `Offset`, `FAST/SLOW`, `Tap Sound`。

```cpp
inline std::wstring Utf8ToWstring(const std::string& str) {
    // UTF-8 -> UTF-16
    int sizeNeeded = MultiByteToWideChar(CP_UTF8, 0, str.c_str(), -1, nullptr, 0);
    std::wstring wstr(sizeNeeded, 0);
    MultiByteToWideChar(CP_UTF8, 0, str.c_str(), -1, &wstr[0], sizeNeeded);
    if (!wstr.empty() && wstr.back() == L'\0') wstr.pop_back();
    return wstr;
}

inline std::string Utf8ToShiftJis(const std::string& utf8Str) {
    // UTF-8 -> Shift-JIS

    int sizeNeeded = WideCharToMultiByte(CP_ACP, 0, wstr.c_str(), -1, nullptr, 0, nullptr, nullptr);
    std::string str(sizeNeeded, 0);
    WideCharToMultiByte(CP_ACP, 0, wstr.c_str(), -1, &str[0], sizeNeeded, nullptr, nullptr);
    if (!str.empty() && str.back() == '\0') str.pop_back();
```

## 17. 楽曲ズレ時のフォールバック（時間同期の二段制御）

### 設計意図
この章の狙いは、BGM再生時刻と譜面進行時刻のズレを実プレイ中に吸収し、体感上の違和感を最小化することです。  
Windows環境では、デバイス負荷・フォーカス遷移・ドライバ状態により、開始直後やプレイ中に数ms〜数十msのズレが発生します。  
このズレを放置すると、見た目と判定タイミングが徐々に乖離し、入力体験の品質が落ちます。  
そこで本作では、ズレ量に応じて補正方法を切り替える「二段制御」を採用しています。  
具体的には、50ms以上の大きなズレは即時追従、小さなズレは補間で徐々に追従する設計です。  
さらに、再生開始前の不安定な時刻帯では不用意に同期を掛けず、開始条件成立後に同期を開始します。  
これにより、復帰時の破綻を避けつつ、通常プレイ時は滑らかな時間進行を維持できます。

### コード抜粋（再生状態に応じた同期入口）

```cpp
// NoteManager.cpp
void NoteManager::SyncTime(float time) {
    float diff = time - this->songTime; // 音声時刻 - 譜面時刻

    // 50msを超えるズレはプレイ感に直結するため即座に合わせる
    if (std::abs(diff) > 0.05f) {
        this->songTime = time;
    }
    else {
        // 音声未開始付近の時刻は補間しない（誤差ノイズを拾わないため）
        if (time <= 0.001f) {
            // no-op
        }
        else {
            // 小ズレは20%ずつ寄せて段差感を抑える
            this->songTime += diff * 0.2f;
        }
    }
}
```

```cpp
float BGMComponent::GetCurrentTime() const {
    if (player.isNull()) return 0.0f; // プレイヤー未生成

    auto* sv = player->GetSourceVoice();
    if (!sv) return 0.0f; // Voice未生成

    XAUDIO2_VOICE_STATE state;
    sv->GetState(&state);

    sv->GetVoiceDetails(&details);

    // SamplesPlayed / SampleRate で現在秒を算出

        return (double)state.SamplesPlayed / (double)details.InputSampleRate;
    }
    return 0.0f;
}
```

### 時間源の優先順位と責務分離
本実装では、時間源を次の2系統に分けて扱います。  
1つ目は `songTime`（譜面進行時刻）で、通常はゲームループ側の `DeltaTime` で前進します。  
2つ目は `audioTime`（音声再生時刻）で、`SamplesPlayed / SampleRate` の実再生基準で取得します。  
重負荷時は `DeltaTime` 側が揺れやすいため、同期時は `audioTime` を正として `songTime` を補正します。  
この「平常時は軽く進める / ズレた時だけ実再生に寄せる」分離で、常時高コストな再計算を避けています。

```cpp
    }
}
```

### 運用上の要点
1. シーン遷移は「ロード対象の非同期実行 + ローディング画面で監視」の二段に固定している。  
2. 譜面生成の重処理は `IngameScene::Init` 側へ閉じ込め、通常プレイフレームへ持ち込まない。  
3. Live2Dモーションはキャッシュ優先 + 非同期I/Oで、同一演出の再発時コストを下げている。  
4. 選曲プレビュー音声など一部は現状同期ロードなので、将来的なジョブ化余地を明示できる構成になっている。

## 20. 最終まとめ（運用・販売に耐えるための実装集合）

### まとめ
本作の実装は、単一機能の最適化ではなく、運用に必要な要素を一貫して揃える方針で構成しています。

- **可変BPM + MBT**: 制作側の配置意図を崩さない
- **120FPS + RawInput**: 判定体験の安定と低遅延
- **局所探索判定 + Instancing**: 高密度譜面でもスケール
- **バイナリ保存 + 互換考慮**: 運用時の実データ保全
- **DirectWrite + 文字コード統一**: 環境差に強いUI基盤
- **時間同期フォールバック**: 再生ズレを閾値ベースで吸収
- **非同期ロード + LoadingScene**: 遷移時の停止感を低減
- **Live2D統合安全策**: 破棄/描画競合とGPU状態汚染を抑制
- **Config + レート + コマンドノーツ + 演出**: 販売後運用に必要な周辺機能を内製
